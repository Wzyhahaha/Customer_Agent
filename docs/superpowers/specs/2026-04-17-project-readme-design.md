# 项目 README 设计

## 背景

当前仓库缺少一份面向接手者的项目级 `README`。从现有代码可以确认，这个项目不是单纯的算法脚本，而是一个可运行的智能客服系统，包含：

- `Streamlit` 页面入口与对话交互
- 基于 `LangChain` 的 ReAct Agent
- 基于 `DashScope` 模型与向量嵌入的 RAG 检索
- 本地 `Chroma` 向量库存储与知识同步
- 客户编号识别、聊天历史持久化、问题单状态流转
- 城市定位、天气上下文、外部报表数据读取
- 独立的 `rag/eval.py` 检索评估脚本

如果 README 只写项目介绍而不写清启动步骤，新同学仍然无法快速运行；如果只堆启动命令，又无法帮助读者理解关键目录、依赖来源和常见故障。这个文档需要同时承担“快速上手”和“最小项目说明”两类职责，但优先级应明确偏向可运行性。

## 目标

- 为新接手项目的同学提供一份可以直接照着执行的 `README`。
- 以 `Windows + PowerShell` 作为主示例，同时给出 `macOS/Linux` 的简短提示。
- 把启动前必须准备的密钥、配置项、外部依赖单独前置说明。
- 覆盖项目入口、核心目录、RAG 评估脚本和常见故障定位方法。
- 对仓库当前不存在的内容保持诚实，例如缺少统一 `requirements.txt` 时要明确说明，而不是伪造文件存在。

## 非目标

- 不把课程设计说明、过程性 spec 文档完整搬进 `README`。
- 不在这次工作中重构项目代码或补齐所有工程化资产。
- 不承诺覆盖所有部署方式；本次只覆盖本地开发/演示运行路径。
- 不把 README 写成面向完全零基础读者的 Python 教程。

## 读者与使用场景

主要读者有两类，但文档要以第一类为主：

- 接手开发或复现实验的同学，需要最短路径跑通项目。
- 老师/评审，需要快速理解项目做什么、包含哪些能力。

对应的使用场景：

- 刚克隆仓库，想知道先装什么、配什么、执行什么命令。
- 启动失败后，想快速判断是依赖问题、密钥问题、定位服务问题还是数据缺失。
- 需要运行 `rag/eval.py` 查看检索质量，而不是只会启动页面。

## 推荐方案

采用“双入口型 README”：

1. 开头先给出快速启动路径，保证读者 5 分钟内可以定位到环境准备、依赖安装、密钥配置和启动命令。
2. 随后再补最小必要的项目说明，包括功能概览、目录结构、RAG 工作流和评估脚本。
3. 文档末尾集中放常见问题，减少主流程被排障内容打断。

这是当前最稳妥的取舍。纯“介绍优先”会拖慢首次运行，纯“命令堆叠”又无法解释外部依赖和目录职责。

## README 信息架构

建议最终 `README.md` 按以下顺序组织：

### 1. 项目简介

用 1 到 2 段文字说明项目定位：

- 这是一个面向扫地机器人/智能客服场景的 Agent + RAG 项目
- 前端入口是 `Streamlit`
- 后端能力包括知识检索、工单处理、定位天气和外部数据查询

简介后紧跟一份简短能力列表，避免一上来就出现大段实现细节。

### 2. 快速开始

这是文档的主入口，应包含：

- 环境要求
- 创建虚拟环境
- 安装依赖
- 配置密钥与服务
- 启动项目
- 最小运行验证

命令示例优先采用 `Windows PowerShell`，例如：

- 创建并激活虚拟环境
- `pip install ...`
- `streamlit run app.py`

同时补充一小段提示，说明其他系统只需要把激活虚拟环境命令替换为对应 shell 语法。

### 3. 启动前检查清单

必须单独成节，显式列出每个关键依赖：

- `DASHSCOPE_API_KEY`
- 高德定位/天气相关密钥
- 网络访问前提
- `data/external/records.csv` 是否存在

每项都要回答三个问题：

- 这个配置是干什么的
- 应该放在哪里或如何提供
- 缺失后会出现什么现象

这一节的目标不是枚举所有环境变量，而是减少“页面打开了但核心能力不能用”的隐性失败。

### 4. 项目结构

只介绍启动和维护需要知道的关键目录，不做全仓库遍历：

- `app.py`
- `agent/`
- `rag/`
- `data/`
- `config/`
- `utils/`
- `chroma_db/`

每个目录用一句话说明职责，帮助读者建立代码地图。

### 5. 核心运行流程

用简短流程说明系统如何工作：

- 页面启动后检查并同步知识库
- Agent 根据消息上下文调用工具
- RAG 根据路由结果检索问题库与结构化知识域
- 聊天记录和问题状态写入本地 JSON

这里不展开实现细节，只帮助读者理解为什么首次启动较慢、为什么某些能力依赖外部服务。

### 6. RAG 评估

明确写出：

- 评估入口是 `rag/eval.py`
- 测试集来自 `data/test_queries.jsonl`
- 输出包括问题库指标、知识域指标、路由准确率和联合命中指标

这一节需要给出运行命令和结果用途，但不需要在 README 中复制评估逻辑代码。

### 7. 常见问题

保留高频且能直接排障的几类：

- 缺依赖导致导入失败
- `DASHSCOPE_API_KEY` 未配置导致模型初始化失败
- 高德密钥或浏览器定位权限问题导致城市/天气不可用
- 首次启动知识库同步较慢
- 外部 CSV 数据缺失导致报告相关能力异常

每个问题尽量写成“现象 -> 原因 -> 处理方式”的格式。

## 依赖说明策略

仓库当前未见统一依赖清单文件。README 需要如实处理这一点：

- 明确说明仓库当前没有 `requirements.txt` 或 `pyproject.toml`
- 给出一份“按现有代码整理的推荐安装包”作为启动参考
- 不把依赖清单伪装成仓库内已经存在的文件

推荐在 README 中把依赖分为两层：

- 核心运行依赖：`streamlit`、`langchain`、`langchain-community`、`chromadb`、`PyYAML`、相关模型 SDK
- 测试/评估依赖：`pytest` 等

如果后续发现仓库中补齐了正式依赖文件，README 再切换成以正式文件为准。

## 错误处理与边界说明

README 需要明确以下边界：

- 没有配置模型密钥时，项目不能正常完成对话和向量化。
- 没有定位服务配置时，定位与天气能力可能失效，但主体界面仍可能启动。
- 首次运行会触发知识同步，耗时高于后续启动是预期现象。
- `rag/eval.py` 的可运行前提是检索相关依赖和模型能力已就绪。

这些说明要放在文档中，而不是依赖读者自己读源代码推断。

## 验收标准

完成后的 `README.md` 应满足以下标准：

- 新读者不用翻代码，能找到启动顺序。
- 所有命令路径与当前仓库结构一致。
- 没有引用仓库中不存在的依赖文件或脚本。
- 能解释 `app.py` 与 `rag/eval.py` 两条主要使用路径。
- 能帮助读者在常见失败场景下快速定位问题。
- 文档内容以实用为主，但仍保留足够的项目概览信息。

## 风险与取舍

- 由于仓库暂未提供正式依赖清单，README 中的安装命令只能基于代码现状给出建议，后续若依赖变动需要同步更新。
- 为了保持实用性，README 不会覆盖所有开发细节，一些深入设计仍应留在 `docs/superpowers/specs/`。
- 以 `Windows` 为主示例能提升当前仓库环境下的可执行性，但跨平台用户仍可能需要自行调整少量命令。
