# 维护保养知识纳入主链路设计

## 1. 背景

当前项目已经完成第一阶段知识重构：

- `policy_rules` 负责保修、免费维修、非保修边界等规则判断
- `troubleshooting_cases` 负责故障现象与低风险排查建议

这套双知识域已经能够稳定支撑高风险客服场景，但原有 `data/policies/维护保养.txt` 中的大量维护保养知识仍未正式进入 typed retrieval 主链路。它们目前要么停留在旧 `policy_answer` 库中，要么仅作为原始素材存在，带来两个直接问题：

- 维护保养类问题无法在新检索链路中被显式识别和评估。
- 继续把维护内容混在旧政策库里，会重新引入知识边界模糊、评估口径混乱的问题。

本设计的目标是在现有 `policy + troubleshooting` 架构上，正式引入第三个知识域 `maintenance`，让维护保养问题进入主检索链路和主评估体系。

## 2. 目标与非目标

### 2.1 目标

- 新增结构化维护知识库 `maintenance_guides`
- 将 `维护保养.txt` 的 200 条内容纳入 typed retrieval 主链路
- 让 `QueryRouter` 显式支持 `maintenance` 与多域 `mixed`
- 让 `rag.eval` 能对维护类样本做单独的路由与 domain 评估
- 保持现有 `policy_rules` 与 `troubleshooting_cases` 结果不被维护类扩展污染

### 2.2 非目标

- 不在本阶段把维护库进一步拆成多个独立 store
- 不在本阶段重构回答层整体提示词体系
- 不在本阶段把所有旧 `policy_answer` 内容全部迁移为结构化知识
- 不在本阶段做复杂分类模型，继续采用轻量规则路由

## 3. 推荐架构

### 3.1 三知识域主链路

主链路从当前的双知识域扩展为三知识域：

- `policy`
  保修、免费维修、第三方配件影响、非保修因素
- `troubleshooting`
  故障现象、排查动作、升级条件
- `maintenance`
  日常清洁、耗材更换、环境适配、长期存放、故障预防

路由结果变为：

- `policy`
- `troubleshooting`
- `maintenance`
- `mixed`
- `other`

### 3.2 mixed 的新定义

`mixed` 不再只表示“保修 + 故障”。

在新设计中，`mixed` 表示用户问题同时命中多个知识域，例如：

- “滤网堵了怎么清理，坏了算保修吗”
  对应 `maintenance + policy`
- “拖布怎么清洗，如果不出水要不要送修”
  对应 `maintenance + troubleshooting`
- “机器人不出水，如果是正常使用坏的还能保修吗”
  对应 `troubleshooting + policy`

这保证 mixed 语义与知识域设计一致，而不是写死为某一种特定组合。

## 4. maintenance_guides 知识库设计

### 4.1 为什么新增单独维护库

不建议继续把维护保养知识放在旧 `policy_answer` 中，原因是：

- 维护问题和保修规则属于不同决策模式
- 维护类回答更强调“怎么做、多久做一次、不要做什么”
- 将其和规则类知识混在一起，会让检索命中后难以控制回答边界

因此维护保养应成为独立知识域，而不是规则库的附属章节。

### 4.2 单库结构化，而不是多库细分

虽然维护保养内部还可以继续拆分为：

- 耗材维护
- 环境适配
- 长期存放
- 故障预防

但第一阶段不建议一开始就拆成多个 store。推荐做法是：

- 新增一个 `maintenance_guides` store
- 设计上覆盖全量维护保养内容
- 导入和验收按三批进行

这样做的优点是：

- 路由复杂度只增加一个知识域
- 评估与上下文格式只增加一条维护线
- 可以先建立稳定边界，再决定是否后续细分

## 5. 维护知识条目 schema

### 5.1 条目原则

每条维护知识不以“章节”为单位，而以“用户可问的问题单元”为单位。

一条记录应只表达一个稳定的维护主题，例如：

- 主刷多久更换一次
- 拖布如何清洗和晾干
- 木地板拖地要注意什么
- 长期不用前要如何存放机器人

不应把多个主题塞进一条超长记录，否则检索命中后仍然难以约束回答。

### 5.2 建议字段

- `id`
- `knowledge_type`
  固定为 `maintenance`
- `section`
  大类，如 `通用基础维护`、`耗材专项维护与更换`、`环境适配维护`
- `topic`
  小主题，如 `主刷更换周期`、`拖布清洗与晾干`
- `applicable_products`
  `扫地机器人`、`扫拖一体`、`通用`
- `user_intents`
  常见问法列表
- `trigger_conditions`
  触发条件，如 `使用后`、`每周`、`宠物家庭`
- `recommended_actions`
  推荐动作
- `do_not_do`
  禁忌动作
- `frequency`
  `每日`、`每周`、`每月`、`1-3个月`、`长期存放前`
- `consumables`
  涉及的耗材或部件
- `escalation_condition`
  超出日常维护、建议售后检测的条件
- `content`
  最终入向量库的正文

### 5.3 section 与 topic 的分层意义

`section` 与 `topic` 必须分开保留。

原因：

- 评估时可以按 `section` 观察大类命中率
- 也可以按 `topic` 观察细粒度准确率
- 生成时可以用 `topic` 约束回答聚焦

如果没有这两层结构，维护评估会再次退化为“命中了哪一大段章节文本”。

### 5.4 escalation_condition 的必要性

维护类问题与故障类问题边界天然接近，例如：

- “滤网堵了怎么清理”
- “拖布老化了还要不要送修”
- “清理后吸力还是很差”

因此每条维护记录都应显式提供何时仍属于维护、何时应转售后检测的条件，避免维护建议误导到高风险诊断。

## 6. 导入策略

虽然目标是“全量纳入”，实现上仍按三批导入同一个 `maintenance_guides` 库：

### 第一批

- 通用基础维护
- 耗材专项维护与更换

### 第二批

- 扫拖一体拖地功能专属维护
- 环境适配维护

### 第三批

- 长期存放维护
- 故障预防维护

这样做不是缩 scope，而是降低一次性导入过大带来的路由、检索和评估噪声。

## 7. 路由设计扩展

### 7.1 新增 maintenance 路由

路由器需要显式新增 `maintenance` 域，用于识别以下问题：

- 耗材更换周期
- 清洁和晾干动作
- 水箱、拖布、滤网、主刷、边刷维护
- 地面材质与使用环境限制
- 长期存放和保养注意事项

### 7.2 mixed 变为多域组合

实现上，路由器不再只返回某个固定混合域，而应支持多个知识域同时命中，最后再对外归一化为：

- 单域命中 -> 对应单域 route
- 多域命中 -> `mixed`
- 无域命中 -> `other`

### 7.3 典型示例

- “拖布怎么清洗，多久换一次”
  -> `maintenance`
- “滤网堵了怎么清理，坏了算保修吗”
  -> `mixed`
- “木地板拖地要注意什么”
  -> `maintenance`
- “机器人不出水，如果是正常使用坏的还能保修吗”
  -> `mixed`

## 8. Typed Retrieval 扩展

### 8.1 RetrievalBundle 扩展

现有 `RetrievalBundle` 需要新增：

- `maintenance_docs`

### 8.2 检索规则

- `policy` -> 只查 `policy_rules`
- `troubleshooting` -> 只查 `troubleshooting_cases`
- `maintenance` -> 只查 `maintenance_guides`
- `mixed` -> 按命中的知识域组合查询
- `other` -> 只保留相似问法召回，不查结构化知识库

### 8.3 Context Formatter 扩展

上下文格式应新增维护段落，例如：

- `## 维护依据`

保持三知识域的上下文边界清晰，避免模型把维护内容误当成故障诊断或保修结论。

## 9. 评估设计扩展

### 9.1 为什么不能继续复用当前 policy section 统计

维护保养不属于规则判断，直接把维护 section 塞进当前 `expected_policy_sections` 会导致：

- 统计语义混乱
- 无法区分是规则库错、排障库错，还是维护库错
- 后续无法定位真实退化点

### 9.2 推荐评估字段

测试样本应继续保留：

- `expected_route`
- `expected_question_refs`

并扩展为 domain-aware 的 section 标注，例如：

- `expected_domain_sections`

形式可以是：

- `policy:保修判断`
- `troubleshooting:故障排查`
- `maintenance:耗材专项维护与更换`

### 9.3 推荐指标

- `route_stage`
  路由准确率
- `question_stage`
  相似问法召回
- `domain_stage`
  不同知识域的命中与精度
- `joint_hit`
  路由和 domain 同时命中的联合指标

### 9.4 样本构成建议

扩充维护评估时，按以下顺序加入样本：

1. 维护单域样本
   - 耗材更换
   - 清洁动作
   - 环境适配
   - 长期存放
2. 跨域 mixed 样本
   - 维护 + 保修
   - 维护 + 故障

不建议一开始就把维护全量样本全部并入，以免评估解释性下降。

## 10. 实施顺序建议

按以下顺序推进，不建议打乱：

1. 新增 `maintenance_guides` 数据与 store 配置
2. 扩展 `QueryRouter` 到 `maintenance`
3. 扩展 `TypedRetrievalService` 与 `RetrievalBundle`
4. 扩展 `context_formatter`
5. 扩展 `rag.eval` 样本 schema 与指标
6. 最后再调整生成层 Prompt

原因是：

- 先有结构化库，路由和检索才有实际落点
- 先稳定检索与评估，再调整生成层，能避免把生成噪声误判为检索退化

## 11. 风险与取舍

### 11.1 风险

- 维护类关键词如果设计过宽，可能会误吸入故障类问题
- 维护条目如果粒度过粗，会重新引入大段文本召回噪声
- 全量纳入后，如果评估样本不分阶段扩充，会影响指标解释性

### 11.2 取舍

本设计选择：

- 单维护库，而不是多维护子库
- 多批导入，而不是一次性扁平迁移
- 多域 mixed，而不是继续写死为保修+故障

这是为了在保持 typed retrieval 清晰边界的同时，把维护保养完整纳入主链路。

## 12. 预期结果

完成后，项目将从当前的“双知识域客服 Agent”扩展为“三知识域客服 Agent”：

- 规则判断
- 故障排查
- 维护保养

这样：

- 维护类问题可以被显式路由和显式评估
- 原有 `policy_rules` 与 `troubleshooting_cases` 不会被重新污染
- 后续若维护类流量继续变大，再考虑拆成多个维护子库也会更自然
